## 🧩 Feature-Sliced Design

Проект следует методологии **Feature-Sliced Design (FSD)**.

### Уровни абстракции

```
src/
├── app/              # Инициализация (роутинг, стили, провайдеры)
├── views/            # Сборка страниц из виджетов (Pages в терминах FSD)
├── widgets/          # Крупные блоки (Header, ApplicationManager, VacancyList)
├── features/         # Бизнес-сценарии (apply-vacancy, profile-edit, vacancy-filter)
├── entities/         # Бизнес-сущности (User, Vacancy, Application)
└── shared/           # Переиспользуемый код (UI-kit, hooks, S3-client, types)
```

## 🔄 Сценарная логика: Жизненный цикл заявки

Система реализует сложный многоуровневый процесс одобрения, особенно для типа «Практика».

### 1. Подача заявки (Студент)
*   При отклике система делает **snapshot** (слепок) текущих данных студента:
    *   Тип практики (учебная/производственная/преддипломная).
    *   Тема проекта.
*   Это гарантирует, что история заявок останется корректной даже после изменения профиля студента в будущем.

### 2. Модерация ВУЗа (для Практик)
*   Заявки на практики сначала попадают в **University Staff Dashboard**.
*   Сотрудник ВУЗа видит сроки, тему проекта и специальность.
*   Компания-партнер **не видит** заявку, пока ВУЗ её не одобрит.
*   При отказе ВУЗ обязан указать причину.

### 3. Рассмотрение Компанией
*   После аппрува ВУЗа (или сразу для работы/стажировок) заявка появляется в **Partner Dashboard**.
*   Работодатель может пригласить студента или отказать.
*   Система автоматически отслеживает **availableSpots** (лимит мест) и закрывает вакансию при достижении квоты.

## 🔍 Система фильтрации и пагинации

### Серверная пагинация
Реализована через URL-параметры (`?page=1`). Это позволяет:
*   Сохранять состояние при перезагрузке.
*   Делиться ссылками на конкретные страницы.
*   Снизить нагрузку на БД и сеть.

### Умные фильтры
*   **Персональный подбор**: Для студентов автоматически включается фильтр «Только для моей специальности».
*   **Логика курса**: Фильтр по курсу работает по принципу «меньше или равно» — студент видит вакансии, подходящие для его уровня и ниже.
*   **Период**: Фильтрация по датам проверяет полное вхождение периода студента в период вакансии (Intersection logic).

## 🔐 Аутентификация и Роли

Система разделена на 4 основные роли:
1.  **Student**: Поиск, фильтрация, управление профилем и резюме, отклики.
2.  **Organization Representative**: Управление вакансиями, просмотр откликов (после модерации ВУЗа), управление профилем компании.
3.  **University Staff**: Модерация заявок на практику, верификация организаций.
4.  **Admin**: Полный доступ ко всем функциям и системным настройкам.

Безопасность обеспечивается на уровне **Middleware** и **Server Actions**, где проверяется не только факт авторизации, но и принадлежность данных конкретной организации или пользователю.

## 📁 Инфраструктура

*   **Database**: PostgreSQL 15 с миграциями через Drizzle Kit.
*   **Migrations**: Автоматически применяются при старте Docker-контейнера (`drizzle-kit push --force`).
*   **Storage**: MinIO (S3-compatible) для хранения PDF-резюме и логотипов компаний.
*   **Validation**: Zod используется для валидации всех форм и данных в Server Actions.
